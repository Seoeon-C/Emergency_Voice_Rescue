# SoundGuard 프로그램 설명서

## 1. 프로젝트 개요

**SoundGuard**는 카메라가 아닌 **소리**를 기반으로 위험 구역의 무단침입과 응급상황을 감지하는 안전 관리 시스템입니다.

산, 밭, 폐공사장, 폐건물, 저수지, 갯벌, 해안가, 군사지역처럼 사람이 상시 감시하기 어렵고, 카메라 설치가 제한적이거나 사각지대가 많은 장소를 대상으로 합니다.

이 시스템은 주변 소리와 사람 말소리를 입력으로 받아 다음 상황을 판단합니다.

| 상황 번호 | 상황 이름 | 설명 |
|---|---|---|
| 상황 0 | 이상없음 | 위험 조건이 감지되지 않은 상태 |
| 상황 1 | 무단침입 | 발소리, 사람 음성 등으로 사람이 위험구역에 들어온 것으로 판단 |
| 상황 2 | 위험 감지 | 비명, 충격음, 구조 요청 음성 등 응급상황 가능성이 있는 상태 |

---

## 2. 사용 기술

| 기능 | 사용 기술 | 역할 |
|---|---|---|
| 환경음 분석 | BEATs | 발소리, 사람 음성, 비명, 충격음 등 주변 소리 분류 |
| 음성 인식 | Whisper API | 사람의 말을 텍스트로 변환 |
| 상황 판단 | Gemini | 환경음 결과, STT 결과, 체류시간을 종합해 상황 0/1/2 판단 |
| 음성 출력 | 고정 TTS 메시지 | 상황별 안내 방송 출력 |
| 상황 기록 | JSONL 로그 | 시간, 위치, 이벤트 정보를 파일에 저장 |
| 상황실 전송 | Webhook | 상황실 서버나 대시보드로 이벤트 전송 |

---

## 3. 전체 동작 흐름

```text
마이크 입력
→ 주변 소리 녹음
→ BEATs로 환경음 분류
→ 사람 존재 여부 판단
→ 위험구역 체류 시간 계산
→ 사람 음성이거나 분류가 불확실하면 Whisper API로 STT 수행
→ Gemini가 상황 0/1/2 판단
→ 상황에 맞는 고정 안내 방송 출력
→ 이벤트 로그 저장
→ 필요 시 상황실로 메시지 전송
```

---

## 4. 주요 기능

### 4.1 환경음 감지

마이크로 일정 시간 동안 주변 소리를 녹음합니다.

기본 설정은 다음과 같습니다.

```env
SAMPLE_RATE=16000
CHUNK_SECONDS=5
```

감지 대상 예시는 다음과 같습니다.

- 발소리
- 사람 말소리
- 비명
- 충격음
- 유리 깨짐
- 배경소음
- 기타 알 수 없는 소리

---

### 4.2 사람 존재 판단

BEATs가 다음과 같은 소리를 감지하면 사람이 위험구역에 있을 가능성이 있다고 판단합니다.

```text
footstep
speech
unknown
```

`unknown`도 포함한 이유는 실제 환경에서는 발소리나 말소리가 정확히 분류되지 않을 수 있기 때문입니다.

---

### 4.3 위험구역 체류시간 판단

프로그램은 사람이 감지된 시점부터 체류 시간을 계산합니다.

```text
5초 이상 체류 → 상황 1: 무단침입 1차 대응
15초 이상 체류 → 상황 1: 무단침입 2차 대응
```

현재 MVP에서는 실제 위치 센서가 없기 때문에, **사람 존재로 추정되는 소리가 연속 감지되는 시간**을 체류시간으로 사용합니다.

실제 서비스로 확장할 경우 다음 센서와 결합할 수 있습니다.

- PIR 센서
- 거리 센서
- GPS
- 라이다
- CCTV
- 출입 게이트 센서

---

## 5. 상황별 판단 로직

### 상황 0: 이상없음

위험 조건이 충족되지 않은 상태입니다.

예시 조건:

```text
배경소음만 감지됨
사람 존재 감지 없음
응급구조신호 없음
위험음 없음
```

출력:

```text
안내 방송 없음
상황실 전송 없음
감시 지속
```

---

### 상황 1: 무단침입

발소리, 사람 음성, 불확실한 사람 존재 소리가 감지되고 일정 시간 이상 체류하면 무단침입으로 판단합니다.

#### 1차 대응

조건:

```text
사람 존재 추정 + 5초 이상 체류
```

동작:

```text
1차 경고 방송 출력
상황 데이터 기록
상황실 전송
```

1차 경고 메시지:

```text
출입이 허가되지 않은 위험 구역입니다. 즉시 안전한 곳으로 이동해 주세요.
```

#### 2차 대응

조건:

```text
사람 존재 추정 + 15초 이상 체류
```

동작:

```text
2차 경고 방송 출력
상황 데이터 기록
위치정보 포함 상황실 전송
상황실 판단 하에 출동 여부 결정
```

2차 경고 메시지:

```text
위험 구역에 계속 머무르고 있습니다. 위치 정보가 상황실로 전송되었습니다. 즉시 퇴장해 주세요.
```

---

### 상황 2: 위험 감지

위험음이나 응급구조신호가 감지되면 상황 2로 판단합니다.

감지 대상 예시:

```text
비명
충격음
유리 깨짐
도와주세요
살려주세요
119 불러줘
사람이 쓰러졌어요
불났어요
갇혔어요
다쳤어요
```

동작:

```text
상황실 긴급 알림 전송
현장 안내 방송 출력
이벤트 로그 저장
```

응급 안내 메시지:

```text
응급 상황이 감지되었습니다. 가능한 경우 안전한 위치로 이동하고 구조 안내를 기다려 주세요.
```

주의할 점은, 이 프로그램은 실제 구조대에 자동 신고하는 것이 아니라 **상황실 판단용 긴급 알림**을 생성하는 구조입니다.

---

## 6. 허가 사용자 인증 기능

위험구역에 정당하게 들어가야 하는 관리자나 작업자를 위해 간단한 인증 기능을 제공합니다.

현재 MVP에서는 콘솔 비밀번호 입력 방식입니다.

동작 방식:

```text
사람 존재 감지
→ 비밀번호 입력 요청
→ 비밀번호가 맞으면 10초 동안 감지 로직 OFF
→ 10초 후 자동으로 감지 재개
```

기본 설정:

```env
AUTH_PASSWORD=1234
AUTH_DISABLE_SECONDS=10
```

실제 시스템에서는 다음 방식으로 확장할 수 있습니다.

- 물리 버튼
- RFID 카드
- QR 인증
- 관리자 앱 인증
- 블루투스 태그
- NFC 태그

## 8. 파일별 역할

### main.py

전체 프로그램을 실행하는 진입점입니다.

주요 역할:

- 마이크 녹음
- 환경음 분류 호출
- STT 호출
- 체류시간 계산
- Gemini 판단 호출
- TTS 출력 호출
- 로그 저장 및 상황실 전송 호출

---

### environmental_sound.py

BEATs 기반 환경음 분류 모듈입니다.

주요 역할:

- BEATs 모델 로드
- 오디오 전처리
- 환경음 분류
- 프로젝트 라벨로 매핑

프로젝트 라벨:

```text
background
footstep
speech
scream
impact
glass_break
unknown
```

BEATs 파일이나 체크포인트가 없으면 fallback 모드로 실행됩니다.  
단, fallback 모드는 실제 분류 성능이 아니라 실행 흐름 확인용입니다.

---

### stt.py

Whisper API를 이용한 음성 인식 모듈입니다.

주요 역할:

- 녹음된 wav 파일을 OpenAI Whisper API로 전송
- 사람 말을 텍스트로 변환
- Gemini 판단 모듈에 STT 결과 전달

---

### decision.py

Gemini 기반 상황 판단 모듈입니다.

주요 역할:

- 환경음 결과 확인
- STT 텍스트 확인
- 체류시간 확인
- 허가 사용자 여부 확인
- 상황 0/1/2 판단
- 어떤 TTS 메시지를 출력할지 결정
- 상황실 전송 여부 결정

---

### output.py

출력과 기록을 담당하는 모듈입니다.

주요 역할:

- 고정 안내 메시지 출력
- mp3 음성 파일 재생
- 이벤트 로그 저장
- 상황실 Webhook 전송

---

### config.py

환경변수를 불러오는 설정 파일입니다.

`.env` 파일의 값을 읽어 프로그램 전체에서 사용합니다.

## 12. 프로그램의 장점

### 12.1 카메라 없이 감지 가능

카메라가 없어도 발소리, 사람 음성, 비명, 충격음으로 위험 가능성을 감지할 수 있습니다.

### 12.2 사각지대 보완 가능

소리는 시야가 막힌 곳에서도 감지될 수 있어 CCTV 사각지대를 보완할 수 있습니다.

### 12.3 상황별 자동 대응 가능

무단침입과 위험 감지를 구분하여 다른 안내 방송과 다른 상황실 메시지를 전송할 수 있습니다.

### 12.4 확장성이 높음

추후 센서, GPS, CCTV, 웹 대시보드, 문자 알림, 지도 표시 기능과 결합할 수 있습니다.

## 13. 한계점

### 13.1 오탐 가능성

바람, 동물, 기계음, 주변 소음이 사람 소리나 위험음으로 잘못 분류될 수 있습니다.

### 13.2 STT 비용과 네트워크 의존성

Whisper API를 사용하기 때문에 인터넷 연결과 API 비용이 필요합니다.

### 13.3 BEATs 세팅 난이도

BEATs는 별도의 `BEATs.py`와 체크포인트가 필요하므로 초기 설정이 다소 복잡합니다.

### 13.4 실제 신고 자동화는 위험

AI 판단만으로 실제 구조대 신고까지 자동화하면 오신고 문제가 발생할 수 있습니다.  
따라서 현재 구조는 상황실 판단 후 출동 여부를 결정하도록 설계되어 있습니다.

## 14. 향후 개선 방향

### 14.1 웹 대시보드 추가

상황실에서 이벤트를 실시간 확인할 수 있는 웹 화면을 만들 수 있습니다.

기능 예시:

- 실시간 이벤트 리스트
- 위치 지도 표시
- 위험도 색상 표시
- 출동 여부 버튼
- 이벤트 상세 로그

### 14.2 센서 결합

소리만으로 판단하지 않고 센서를 함께 사용하면 정확도가 높아집니다.

추천 센서:

- PIR 센서
- 초음파 센서
- GPS
- 라이다
- 열화상 센서

### 14.3 자체 데이터셋 학습

발소리, 비명, 충격음, 사람 음성 등 프로젝트 환경에 맞는 데이터를 수집해 fine-tuning하면 정확도가 향상됩니다.

### 14.4 모바일 알림

상황실뿐 아니라 관리자 휴대폰으로 알림을 보낼 수 있습니다.

예:

- 카카오톡 알림
- 문자 알림
- 이메일 알림
- 앱 푸시 알림

## 15. 발표용 한 줄 설명

> SoundGuard는 주변 소리와 사람 음성을 분석해 출입 제한 위험구역의 무단침입과 응급상황을 감지하고, 현장 경고 방송과 상황실 알림을 자동으로 수행하는 음향 기반 안전 관리 시스템입니다.

## 16. 발표용 핵심 차별점

기존 보안 시스템은 CCTV 중심이지만, SoundGuard는 **소리**를 중심으로 위험을 감지합니다.

따라서 다음과 같은 환경에서 강점이 있습니다.

- 카메라 설치가 어려운 장소
- 야간이나 안개 등 시야가 좋지 않은 장소
